Configurar observabilidade no Grafana

Tutorial passo a passo para subir Prometheus, Loki, Promtail e Grafana no SoftMusic, validar métricas e logs, montar dashboards úteis e configurar alertas — local e produção.

Grafana 11 Prometheus 2.53 Loki 3.1 Docker Compose

Arquitetura da observabilidade

O SoftMusic usa o perfil Docker observability. Os apps enviam telemetria; o Grafana centraliza visualização.

Aplicações (perfil app)

API expõe /metrics (prom-client) · Python AI expõe /metrics (Prometheus client) · variáveis OTEL_EXPORTER_OTLP_ENDPOINT apontam para o collector · logs em stdout → Docker → Promtail

Prometheus

Coleta métricas a cada 15s. Retenção 15 dias. Regras em alerts.yml.

Loki + Promtail

Promtail lê logs dos containers via Docker socket e envia ao Loki (retenção 7 dias).

Grafana

UI em :3000. Data sources provisionados automaticamente (Prometheus + Loki).

OTEL Collector

Recebe traces/métricas OTLP (:4317/:4318). Base para evolução futura de APM.

1. Subir a stack de observabilidade

Pré-requisito O perfil observability é independente, mas para métricas dos apps você precisa do perfil app rodando (API + Python AI).

Local (desenvolvimento)

PowerShell / Bash 
cd softmusic

# Infra + apps + observabilidade
docker compose -f infra/docker/docker-compose.yml \
  --profile infra --profile app --profile observability \
  up -d --build

# Só observabilidade (apps já rodando)
docker compose -f infra/docker/docker-compose.yml \
  --profile observability up -d

Variáveis no infra/docker/.env

VariávelExemplo localDescrição
GRAFANA_ADMIN_USERadminUsuário admin do Grafana
GRAFANA_ADMIN_PASSWORDsenha forteObrigatória — compose falha sem ela
GRAFANA_ROOT_URLhttp://localhost:3000URL pública (links em alertas/e-mails)
GRAFANA_PORT3000Porta exposta no host
PROMETHEUS_PORT9090UI do Prometheus
OTEL_EXPORTER_OTLP_ENDPOINThttp://otel-collector:4317Já definido nos serviços app

Verificação rápida

Shell 
docker ps --format "table {{.Names}}\t{{.Status}}" | findstr softmusic-grafana
curl -s -o NUL -w "Grafana:%{http_code} " http://localhost:3000/login
curl -s -o NUL -w "Prometheus:%{http_code} " http://localhost:9090/-/healthy
curl -s http://localhost:8080/metrics | findstr process_cpu
curl -s http://localhost:8000/metrics | findstr python_info

2. Primeiro acesso ao Grafana

  1. Abra http://localhost:3000.
  2. Entre com GRAFANA_ADMIN_USER / GRAFANA_ADMIN_PASSWORD do seu .env.
  3. Na primeira visita o Grafana pode pedir para alterar a senha — em dev você pode pular; em produção altere imediatamente.
  4. Vá em Connections → Data sources. Você deve ver: Prometheus (default) Loki — provisionados por arquivo, sem clique manual.
  5. Clique em Prometheus → Save & test. Deve aparecer Successfully queried the Prometheus API.
  6. Repita em Loki → Save & test.
Dica Os data sources estão em infra/monitoring/grafana/provisioning/datasources/datasources.yml e são montados read-only no container. Para mudar URL ou auth, edite o arquivo e reinicie o Grafana.

3. Data sources (já provisionados)

Não é necessário criar manualmente — mas entender o mapeamento ajuda em produção:

datasources.yml (resumo) 
Prometheus → http://prometheus:9090   (default)
Loki       → http://loki:3100

Testar no Explore

  1. Menu lateral → Explore
  2. Selecione Prometheus
  3. Query: up → Run query
  4. Troque para Loki
  5. Query: {service=~"softmusic-.*"} → Run query

Você deve ver séries up{job="softmusic-api"}, softmusic-python-ai, etc., e linhas de log dos containers.

4. Criar dashboards

A pasta SoftMusic é provisionada em infra/monitoring/grafana/provisioning/dashboards/dashboards.yml. Salve JSONs em infra/monitoring/grafana/dashboards/ para versionar no Git — ou crie pela UI e exporte depois.

Pasta vazia? Se não houver JSON em dashboards/, crie o primeiro painel pela UI: Dashboards → New → New dashboard. Ao exportar, coloque o arquivo na pasta acima e reinicie o Grafana.

4.1 Painel — Visão geral da plataforma

Dashboards → New dashboard → Add visualization

Stat — Serviços UP
sum(up{job=~"softmusic-.*|prometheus"})
Time series — CPU API
rate(process_cpu_seconds_total{job="softmusic-api"}[5m])
Time series — Memória Python AI
process_resident_memory_bytes{job="softmusic-python-ai"}
Stat — Targets down
count(up == 0) OR vector(0)
  1. Organize em uma linha de Stat (UP) + linha de Time series (recursos).
  2. Salve como SoftMusic / Overview.
  3. Defina refresh 30s e intervalo de tempo Last 6 hours.

4.2 Painel — API & Python AI

Métricas default do prom-client (Node) e prometheus_client (Python) já estão disponíveis:

Node — Event loop lag (API)
nodejs_eventloop_lag_seconds{job="softmusic-api"}
Node — Heap usado (API)
process_heap_bytes{job="softmusic-api"}
Python — Info
python_info{job="softmusic-python-ai"}
Disponibilidade
avg_over_time(up{job=~"softmusic-api|softmusic-python-ai"}[1h])
Evolução recomendada Adicione métricas HTTP customizadas (http_requests_total, latência por rota) no BFF para alimentar os alertas já definidos em alerts.yml. Hoje o endpoint /metrics expõe métricas de runtime do processo.

4.3 Painel — Logs com Loki

  1. Novo painel → data source Loki → tipo Logs.
  2. Query base:
LogQL 
{service=~"softmusic-api|softmusic-python-ai|softmusic-worker"}

Filtros úteis:

LogQL — exemplos 
# Erros na API
{service="softmusic-api"} |= "error" or |= "ERROR"

# Worker / análise
{service="softmusic-worker"} |= "run_analysis"

# Scheduler billing
{service="softmusic-scheduler"}

# Por container
{container="softmusic-python-ai"}

Adicione um painel Logs volume com:

sum by (service) (count_over_time({service=~"softmusic-.*"}[5m]))

Variáveis de dashboard (opcional)

Dashboard settings → Variables → Add variable

  • Nome: service
  • Tipo: Query (Loki)
  • Query: label_values(service)
  • Use nas queries: {service="$service"}

5. Alertas

Existem dois níveis: regras no Prometheus (arquivo) e contact points no Grafana (UI).

5.1 Regras já definidas (alerts.yml)

AlertaCondiçãoStatus
ServiceDown up == 0 por 2 min Funciona hoje
HighErrorRate 5xx > 1% em 5 min Requer métrica HTTP
AnalysisQueueBacklog fila Celery > 100 Requer métrica fila

Verifique em Prometheus → Alerts e Status → Rules.

5.2 Alertas no Grafana (recomendado para notificação)

  1. Alerting → Contact points → New — configure e-mail, Slack ou Telegram.
  2. Notification policies — rota default para o contact point.
  3. Em um painel, edite um painel → Alert → crie regra, ex.:
PromQL — API down 
up{job="softmusic-api"} == 0

Condição: IS BELOW 1 por 2m → severidade critical → notificar contact point.

5.3 Silêncios e manutenção

Em deploys planejados: Alerting → Silences → New silence com matcher job=softmusic-api e duração do deploy.

6. Prometheus — targets e scrape

Configuração: infra/monitoring/prometheus/prometheus.yml

JobTargetPath
prometheuslocalhost:9090/metrics
softmusic-apiapi:8080/metrics
softmusic-python-aipython-ai:8000/metrics
rabbitmqrabbitmq:15692plugin Prometheus

Em localhost:9090/targets todos devem estar UP.

RabbitMQ DOWN? O plugin de métricas precisa estar habilitado na imagem/config do RabbitMQ. Se não usar, remova o job rabbitmq do prometheus.yml ou ignore o target.

Recarregar config sem reiniciar

curl -X POST http://localhost:9090/-/reload

(Requer --web.enable-lifecycle — já ativo no compose.)

7. Loki & Promtail

Promtail descobre containers via Docker socket e envia logs com labels container e service (regex softmusic-*).

Validar ingestão

# Labels disponíveis
curl -s "http://localhost:3100/loki/api/v1/labels" | jq .

# Últimos streams do serviço API
curl -sG "http://localhost:3100/loki/api/v1/query" \
  --data-urlencode 'query={service="softmusic-api"}' \
  --data-urlencode 'limit=5'

Retenção

7 dias em loki-config.yaml (retention_period: 168h). Ajuste conforme disco da VPS.

Windows + Docker Desktop Promtail precisa do socket /var/run/docker.sock montado. No Docker Desktop WSL2 isso funciona; se logs não aparecerem, confira se o container promtail está healthy e se os containers têm prefixo softmusic-.

8. OpenTelemetry Collector

O collector sobe com o perfil app e observability. Apps já têm:

OTEL_EXPORTER_OTLP_ENDPOINT=http://otel-collector:4317

Pipeline atual (otel-collector-config.yaml):

  • Traces → export logging (dev)
  • Metrics OTLP → exporter Prometheus na porta 8889 do collector

Para visualizar métricas OTLP no Grafana, adicione scrape no Prometheus:

prometheus.yml (sugestão) 
- job_name: otel-collector
  static_configs:
    - targets: ["otel-collector:8889"]

9. Produção

9.1 Variáveis

Use infra/docker/.env.production.example:

GRAFANA_ROOT_URL=https://grafana.softmusic.com.br
GRAFANA_ADMIN_PASSWORD=<senha-forte-única>
PROMETHEUS_PORT=9090   # não exponha publicamente sem auth

9.2 Expor Grafana com segurança

  1. Coloque Grafana atrás do Nginx com TLS (Let's Encrypt).
  2. Restrinja Prometheus/Loki à rede interna Docker — só Grafana acessa.
  3. Desative signup: já está GF_USERS_ALLOW_SIGN_UP=false.
  4. Crie usuários read-only para a equipe (Role: Viewer).
  5. Considere SSO (Google/GitHub) via Grafana auth em produção.

9.3 Backup

  • Volume softmusic_grafana_data — dashboards criados na UI, usuários, alertas.
  • Volume softmusic_prometheus_data — histórico de métricas.
  • Prefira dashboards versionados em infra/monitoring/grafana/dashboards/*.json.

9.4 Subir em produção

docker compose -f infra/docker/docker-compose.yml \
  -f infra/docker/docker-compose.prod.yml \
  --env-file /opt/softmusic/.env.production \
  --profile infra --profile app --profile observability \
  up -d prometheus grafana loki promtail otel-collector

10. Troubleshooting

SintomaCausa provávelSolução
Grafana sem data sources Volume provisioning não montado Verifique bind ../monitoring/grafana/provisioning e reinicie
Prometheus target API DOWN API não está no mesmo network Suba perfil app; confira docker network inspect softmusic-network
Loki vazio no Explore Promtail sem acesso ao Docker socket Logs do softmusic-promtail; confira permissões do socket
Alertas HighErrorRate nunca disparam Métrica http_requests_total ausente Instrumente o BFF ou use alerta up == 0 por enquanto
Login Grafana falha após mudar .env Senha só aplicada na 1ª inicialização Reset via grafana cli admin reset-admin-password no container
Links em e-mails apontam localhost GRAFANA_ROOT_URL incorreto Ajuste para URL pública e reinicie Grafana

Reset senha admin (emergência)

docker exec -it softmusic-grafana grafana cli admin reset-admin-password 'NovaSenhaForte123!'
