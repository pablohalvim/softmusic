Deploy SoftMusic em VPS — Jenkins em container + Docker Compose

Cenário: uma VPS única onde o Jenkins roda dentro de um container e fala com o daemon Docker do host via /var/run/docker.sock. O deploy é local, sem SSH: as imagens são buildadas no daemon do host (sem push para registry) e o docker compose sobe os serviços na própria máquina. Mesmo padrão do projeto sportshub que já roda nesta VPS.

Jenkins em container Sem SSH Sem registry Secret text 5 pipelines
Convenções deste guia
  • VPS única — Docker Engine, container do Jenkins, MySQL, Redis, RabbitMQ, API, IA, frontends e observabilidade na mesma máquina.
  • Container do Jenkins — usuário jenkins (UID 1000), com --group-add do GID do grupo docker do host e o socket montado.
  • Projeto compose softmusic — todos os serviços compartilham o projeto e a rede softmusic-network.
  • DEPLOY_DIR — pasta (no volume do Jenkins) onde os pipelines copiam compose + configs + .env.production.
  • Sem segredos no código — os pipelines usam credenciais Secret text e o render-env.sh monta o .env.

Domínios de produção

O SoftMusic expõe três portais públicos. Configure DNS (A/AAAA ou CNAME) apontando para o IP da VPS.

Domínios no .env WEB_ORIGIN, API_BASE_URL, LP_ORIGIN e ADMIN_ORIGIN têm defaults em render-env.sh; sobrescreva por Environment variables do job se seus domínios diferirem.

Por que copiar os configs para o DEPLOY_DIR? Como o Jenkins fala com o daemon do host, todo bind mount (observabilidade, nginx, mysql/init) precisa apontar para um caminho que o host enxergue. Os pipelines copiam esses arquivos para o DEPLOY_DIR (dentro do volume do Jenkins, visível no host em DEPLOY_DIR_HOST) e os bind mounts usam esse caminho.

Pré-requisito — container do Jenkins com acesso ao Docker

O usuário jenkins (UID 1000) precisa falar com o Docker do host. Recrie o container com o socket, o binário do Docker e o --group-add do GID do grupo docker:

bash — na VPS 
DOCKER_GID=$(getent group docker | cut -d: -f3)

docker run -d \
  --name jenkins \
  --restart unless-stopped \
  -p 8080:8080 -p 50000:50000 \
  -v /dados/jenkins_home:/var/jenkins_home \
  -v /var/run/docker.sock:/var/run/docker.sock \
  -v /usr/bin/docker:/usr/bin/docker \
  --group-add "$DOCKER_GID" \
  jenkins/jenkins:lts

# Se usar docker compose no pipeline, monte o plugin (o caminho pode variar):
#   -v /usr/libexec/docker/cli-plugins/docker-compose:/usr/local/lib/docker/cli-plugins/docker-compose

Teste dentro do container antes de rodar os jobs:

bash 
docker exec -u jenkins jenkins docker ps
docker exec -u jenkins jenkins docker compose version
GPU (opcional) Se a IA usa GPU, instale o NVIDIA Container Toolkit no host. Sem GPU, ajuste a variável DOCKER_GPUS do serviço python-ai conforme necessário.

DEPLOY_DIR e bind mounts

Os pipelines gravam compose + configs em DEPLOY_DIR. Como esse diretório fica no volume do Jenkins (/dados/jenkins_home no host), os arquivos existem também no host — e é esse caminho (DEPLOY_DIR_HOST) que os bind mounts usam.

VariávelVisão do Jenkins (container)No host
DEPLOY_DIR/var/jenkins_home/deploy/softmusic/dados/jenkins_home/deploy/softmusic
DEPLOY_DIR_HOST—/dados/jenkins_home/deploy/softmusic
Se o jenkins_home estiver em outro caminho no host Ajuste DEPLOY_DIR_HOST (em Environment variables do job ou no environment {} do Jenkinsfile). Se ficar errado, observabilidade/nginx sobem com config vazia.

Como os pipelines usam as credenciais

groovy — trecho comum 
environment {
  DEPLOY_DIR      = "${env.DEPLOY_DIR ?: '/var/jenkins_home/deploy/softmusic'}"
  DEPLOY_DIR_HOST = "${env.DEPLOY_DIR_HOST ?: '/dados/jenkins_home/deploy/softmusic'}"
  IMAGE_TAG       = "${env.BUILD_NUMBER ?: 'latest'}"
  IMAGE_PREFIX    = 'softmusic'
}

stage('Gerar .env de produção') {
  steps {
    withCredentials([
      string(credentialsId: 'softmusic-mysql-root-password',    variable: 'MYSQL_ROOT_PASSWORD'),
      string(credentialsId: 'softmusic-mysql-password',         variable: 'MYSQL_PASSWORD'),
      string(credentialsId: 'softmusic-redis-password',         variable: 'REDIS_PASSWORD'),
      string(credentialsId: 'softmusic-rabbitmq-password',      variable: 'RABBITMQ_PASSWORD'),
      string(credentialsId: 'softmusic-jwt-private-key',        variable: 'JWT_PRIVATE_KEY'),
      string(credentialsId: 'softmusic-grafana-admin-password', variable: 'GRAFANA_ADMIN_PASSWORD'),
      // + admin-jwt, admin-bootstrap, asaas...
    ]) {
      sh 'bash infra/docker/scripts/render-env.sh'
    }
  }
}
  • Apps buildam local: docker build -f infra/docker/Dockerfile.<svc> -t softmusic/<svc>:latest . (sem push).
  • render-env.sh grava DEPLOY_DIR/.env.production; deploy-*.sh copia os assets e sobe o compose.
  • Nenhum segredo no código — apenas IDs de credential Secret text.

Guia de escolha — MySQL 8.4 vs 5.7 (legacy)

CenárioJobMySQL
VPS nova, CPU x86_64 moderna (AVX2)softmusic-infra8.4 (padrão)
CPU antiga, erro ao subir MySQL 8.xsoftmusic-infra-legacy5.7 (LEGACY_MYSQL=1)
Escolha uma vez Rode apenas um dos jobs de infra. Não alterne entre 8.4 e 5.7 no mesmo volume de dados sem migração planejada.

Passo 3 — Ordem do deploy

1. infra ou infra-legacy → 2. ia (migrations) → 3. api → 4. web
  1. softmusic-infra (ou -legacy) — provisiona MySQL + Redis + RabbitMQ + toda a observabilidade.
  2. softmusic-ia — builda a imagem, sobe python-ai + worker e aplica as migrations (alembic upgrade head no entrypoint).
  3. softmusic-api — builda e sobe a API.
  4. softmusic-web — builda e sobe web + landing page (+ nginx).
Regra de ouro sobre migrations O banco só é migrado pelo job softmusic-ia. A API não aplica migrations. Se um deploy depende de schema novo, rode a IA antes.

O que cada job de app faz (local, sem registry)

  1. checkout scm → clona o repo no workspace do Jenkins.
  2. docker build da imagem no daemon do host, marcando :latest (sem push).
  3. render-env.sh gera DEPLOY_DIR/.env.production a partir das credenciais.
  4. deploy-*.sh copia compose + configs para DEPLOY_DIR e roda docker compose up -d --no-deps --force-recreate <serviço> + health check.
admin-web Não é deployado pelas pipelines atuais. Para publicá-lo, crie um Jenkinsfile.admin no mesmo molde e rode o deploy-web.sh com DEPLOY_ADMIN_WEB=1.

Observabilidade

Os dois jobs de infra sobem a stack completa: Prometheus, Loki, Promtail, Grafana e OpenTelemetry Collector (WITH_OBSERVABILITY=1). O deploy-infra.sh verifica no fim se todos os containers subiram.

bash — verificar na VPS 
docker ps \
  --filter name=softmusic-prometheus --filter name=softmusic-loki \
  --filter name=softmusic-promtail --filter name=softmusic-grafana \
  --filter name=softmusic-otel-collector \
  --format 'table {{.Names}}\t{{.Status}}'

Grafana atende em GRAFANA_ROOT_URL. Guia dedicado: docs/producao/tutorial-grafana-observabilidade.html.

TLS / NGINX

O overlay docker-compose.prod.yml inclui nginx (80/443) e um certbot sidecar. Na primeira vez, emita os certificados (DNS já apontando para a VPS):

bash — na VPS 
docker run --rm \
  -v softmusic_certbot_certs:/etc/letsencrypt \
  -v softmusic_certbot_www:/var/www/certbot \
  certbot/certbot certonly --webroot -w /var/www/certbot \
  -d softmusic.com.br -d app.softmusic.com.br -d admin.softmusic.com.br \
  --email voce@dominio.com --agree-tos --no-eff-email

Enquanto não houver certificado, o deploy-web.sh apenas avisa (não falha) que o nginx não subiu.

Rollback

Sem registry, o rollback é rebuildar a imagem a partir do commit anterior — rode o job apontando para o ref anterior. As imagens antigas ficam no daemon do host, marcadas por BUILD_NUMBER, até o docker image prune.

bash — reusar imagem antiga (ex.: build #7) 
cd /dados/jenkins_home/deploy/softmusic
sed -i 's|^SOFTMUSIC_API_IMAGE=.*|SOFTMUSIC_API_IMAGE=softmusic/api:7|' .env.production
docker compose -f docker-compose.yml -f docker-compose.prod.yml \
  --env-file .env.production --profile infra --profile app \
  up -d --no-deps --force-recreate api

Troubleshooting

SintomaCausa provávelAção
permission denied no docker.sock Jenkins sem --group-add do GID do docker Recriar o container do Jenkins (seção Pré-requisito)
Observabilidade/nginx com config vazia DEPLOY_DIR_HOST não bate com o caminho real no host Ajustar DEPLOY_DIR_HOST ao mapeamento do jenkins_home
Deploy infra aborta citando variável Credencial Secret text ausente/vazia Cadastrar a credencial e re-rodar
image not found no compose up Job de app não buildou antes Rodar o job de app (ele builda e sobe)
Coluna/tabela nova não existe Migration não aplicada Rodar o job softmusic-ia
MySQL em restart loop CPU incompatível com MySQL 8.4 Usar softmusic-infra-legacy
502 nos domínios nginx sem certificado / app não subiu Emitir TLS; docker logs softmusic-nginx
