Tutorial: Deploy SoftMusic SaaS em VPS

Implante o SoftMusic em produção com Jenkins (build/push de imagens), SSH (deploy remoto) e Docker Compose na VPS. Segredos ficam nas Credentials do Jenkins — nunca no Jenkinsfile.

Jenkins + SSH Docker Hub 5 pipelines Ubuntu 22.04 / 24.04 SaaS multi-domínio
Convenções deste guia
  • VPS de produção — uma máquina Ubuntu com Docker, NGINX (80/443), MySQL, Redis, RabbitMQ, API, IA e frontends.
  • VPS-Jenkins — pode ser a mesma VPS ou outra máquina; usuário jenkins faz SSH como deploy.
  • deploy — usuário Linux na VPS de produção; membro do grupo docker.
  • /opt/softmusic/repo — clone Git do monorepo SoftMusic.
  • /opt/softmusic/.env.production — arquivo de ambiente enviado pelo Jenkins a cada deploy.
  • Sem segredos no código — pipelines usam apenas credentials('...').

Domínios de produção

O SoftMusic SaaS expõe três portais públicos. Configure DNS (A/AAAA ou CNAME via Cloudflare) apontando para o IP da VPS.

DomínioAppPúblico
softmusic.com.br apps/lp Landing page, planos, botão Login
app.softmusic.com.br apps/web + API em /api Cadastro, bandas, biblioteca, cifras, faturas (owner)
admin.softmusic.com.br apps/admin-web Equipe SoftMusic (credenciais internas)
Variáveis no .env de produção Defina WEB_ORIGIN, API_BASE_URL, LP_ORIGIN e ADMIN_ORIGIN com esses domínios HTTPS. Modelo completo em infra/docker/.env.production.example.

Detalhes de produto, planos e billing: docs/arquitetura/saas-bandas-licencas.md.

Parte 1 — Preparar a VPS de produção

Execute como root ou com sudo. Recomendado: Ubuntu 22.04 ou 24.04 LTS. Mínimo sugerido: 8 vCPU, 32 GB RAM, 200 GB SSD; GPU NVIDIA se rodar IA na mesma máquina.

1.1 Atualizar o sistema

bash 
apt update && apt upgrade -y
apt install -y ca-certificates curl gnupg lsb-release ufw fail2ban git unattended-upgrades

dpkg-reconfigure -plow unattended-upgrades
timedatectl set-timezone America/Sao_Paulo
hostnamectl set-hostname softmusic-prod-01

1.2 Hardening básico de SSH

Edite /etc/ssh/sshd_config:

sshd_config 
PermitRootLogin no
PasswordAuthentication no
PubkeyAuthentication yes
ChallengeResponseAuthentication no
Atenção Só desligue PasswordAuthentication depois de validar login com chave (passo 2.3).
bash 
systemctl restart ssh

1.3 Firewall (UFW) — portas 80 e 443

O NGINX na VPS expõe HTTP/HTTPS para os três domínios. MySQL, Redis e RabbitMQ ficam apenas na rede Docker interna.

bash — VPS produção 
ufw default deny incoming
ufw default allow outgoing
ufw allow OpenSSH
ufw allow 22/tcp
ufw allow 2230/tcp              # SSH customizado (se usar)
ufw allow 80/tcp comment 'HTTP — redirect / ACME'
ufw allow 443/tcp comment 'HTTPS — LP, app, admin'

ufw enable
ufw status verbose
Não exponha portas 3306, 6379 ou 5672 na internet. O docker-compose.prod.yml já evita bind público desses serviços.

1.4 Instalar Docker Engine + Compose

bash 
install -m 0755 -d /etc/apt/keyrings
curl -fsSL https://download.docker.com/linux/ubuntu/gpg \
  | gpg --dearmor -o /etc/apt/keyrings/docker.gpg
chmod a+r /etc/apt/keyrings/docker.gpg

echo \
  "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.gpg] \
  https://download.docker.com/linux/ubuntu \
  $(. /etc/os-release && echo "$VERSION_CODENAME") stable" \
  > /etc/apt/sources.list.d/docker.list

apt update
apt install -y docker-ce docker-ce-cli containerd.io docker-buildx-plugin docker-compose-plugin
systemctl enable --now docker
docker --version
docker compose version

1.5 Criar o usuário deploy

bash 
adduser --disabled-password --gecos "" deploy
usermod -aG docker deploy

mkdir -p /opt/softmusic
chown deploy:deploy /opt/softmusic

mkdir -p /home/deploy/.ssh
chmod 700 /home/deploy/.ssh
touch /home/deploy/.ssh/authorized_keys
chmod 600 /home/deploy/.ssh/authorized_keys
chown -R deploy:deploy /home/deploy/.ssh

1.6 Clonar o repositório em /opt/softmusic/repo

Faça o clone antes do primeiro deploy Jenkins — os pipelines fazem git fetch e reset --hard origin/main, mas exigem o diretório .git existente.

bash — como deploy 
sudo -iu deploy
cd /opt/softmusic
git clone https://github.com/SEU_ORG/softmusic.git repo
cd repo && git checkout main
ls -la infra/jenkins/
Deploy key ou PAT Se o repositório for privado, configure chave de deploy do usuário deploy ou use HTTPS com token no git clone.

Parte 2 — Jenkins e chave SSH

Jenkins instalado via apt, usuário jenkins, home em /var/lib/jenkins. Para docker build e push, o usuário jenkins precisa estar no grupo docker.

2.0 Instalar o Jenkins

bash — VPS-Jenkins 
apt update
apt install -y ca-certificates curl gnupg fontconfig openjdk-17-jre

curl -fsSL https://pkg.jenkins.io/debian-stable/jenkins.io-2023.key \
  | tee /usr/share/keyrings/jenkins-keyring.asc > /dev/null
echo "deb [signed-by=/usr/share/keyrings/jenkins-keyring.asc] \
      https://pkg.jenkins.io/debian-stable binary/" \
  > /etc/apt/sources.list.d/jenkins.list

apt update
apt install -y jenkins
systemctl enable --now jenkins
cat /var/lib/jenkins/secrets/initialAdminPassword

UI na porta 8080. Instale plugins: Pipeline, SSH Agent, Credentials Binding.

bash 
usermod -aG docker jenkins
systemctl restart jenkins

2.1 Gerar chave ed25519 para deploy

bash — VPS-Jenkins 
sudo -iu jenkins
mkdir -p ~/.ssh && chmod 700 ~/.ssh
ssh-keygen -t ed25519 -C "jenkins@softmusic-ci" -f ~/.ssh/id_ed25519_deploy -N ""
ls -la ~/.ssh/id_ed25519_deploy*
ArquivoUso
id_ed25519_deploy Chave privada → credential softmusic-ssh-deploy-vps
id_ed25519_deploy.pub Chave pública → /home/deploy/.ssh/authorized_keys na VPS

2.2 Copiar chave pública para a VPS

  1. Na VPS-Jenkins: sudo cat /var/lib/jenkins/.ssh/id_ed25519_deploy.pub
  2. Na VPS de produção: cole em /home/deploy/.ssh/authorized_keys
  3. Ajuste permissões: 
    chown deploy:deploy /home/deploy/.ssh/authorized_keys
chmod 600 /home/deploy/.ssh/authorized_keys

2.3 Testar SSH sem senha

bash 
sudo -iu jenkins
ssh -i ~/.ssh/id_ed25519_deploy -p 22 -o StrictHostKeyChecking=accept-new \
    deploy@SEU_IP_VPS 'docker version && id && ls /opt/softmusic/repo/.git'

Esperado: versão do Docker, deploy no grupo docker e diretório .git presente.

Parte 3 — Credenciais no Jenkins

Manage Jenkins → Credentials → System → Global credentials → + Add Credentials

Documentação detalhada: infra/jenkins/credentials.md

Regra de ouro Nunca coloque senhas, tokens ou IPs no Jenkinsfile. Use apenas credentials('id-exato').

As 5 credenciais obrigatórias

IDs devem bater letra por letra ou o pipeline falha com CredentialsNotFoundException.

softmusic-docker-registry-creds
Username with password

Docker Hub — build/push das imagens (api, ia, web, lp)

softmusic-ssh-deploy-vps
SSH Username with private key

Username: deploy — todos os pipelines

softmusic-deploy-host
Secret text

IP ou DNS da VPS (sem http://)

softmusic-deploy-host-port
Secret text

Porta SSH (ex.: 22 ou 2230)

softmusic-production-env
Secret file

Arquivo .env completo de produção

#IDKindUso
1softmusic-docker-registry-credsUsername with passwordLogin Docker Hub nos jobs api, ia, web
2softmusic-ssh-deploy-vpsSSH private keySSH/SCP para usuário deploy
3softmusic-deploy-hostSecret textHost da VPS
4softmusic-deploy-host-portSecret textPorta SSH
5softmusic-production-envSecret fileCopiado para /opt/softmusic/.env.production

3.1 Arquivo softmusic-production-env

Baseie-se em infra/docker/.env.production.example. Inclua obrigatoriamente:

  • MySQL, Redis, RabbitMQ
  • JWT e URLs dos domínios (WEB_ORIGIN, LP_ORIGIN, ADMIN_ORIGIN)
  • Asaas (ASAAS_API_KEY, ASAAS_WEBHOOK_TOKEN)
  • Tags de imagem (SOFTMUSIC_*_IMAGE) — pipelines de app atualizam automaticamente

3.2 Teste rápido (job Pipeline descartável)

groovy 
pipeline {
  agent any
  environment {
    DEPLOY_HOST     = credentials('softmusic-deploy-host')
    DEPLOY_SSH_PORT = credentials('softmusic-deploy-host-port')
    DOCKER_CREDS    = credentials('softmusic-docker-registry-creds')
  }
  stages {
    stage('Registry') {
      steps {
        sh 'echo "$DOCKER_CREDS_PSW" | docker login -u "$DOCKER_CREDS_USR" --password-stdin'
        sh 'docker logout || true'
      }
    }
    stage('SSH') {
      steps {
        sshagent(credentials: ['softmusic-ssh-deploy-vps']) {
          sh '''
            ssh -o StrictHostKeyChecking=no -p $DEPLOY_SSH_PORT deploy@$DEPLOY_HOST \
              'docker version --format "{{.Server.Version}}" && id'
          '''
        }
      }
    }
  }
}

Parte 4 — Jenkinsfiles em infra/jenkins/

Crie um job Pipeline por Jenkinsfile, apontando para o caminho no repositório (Multibranch ou SCM).

Job JenkinsJenkinsfileDescrição
softmusic-infra infra/jenkins/Jenkinsfile.infra MySQL 8.4 + Redis + RabbitMQ + observabilidade
softmusic-infra-legacy infra/jenkins/Jenkinsfile.infra-legacy MySQL 5.7 para CPUs antigas
softmusic-ia infra/jenkins/Jenkinsfile.ia python-ai + worker Celery (GPU)
softmusic-api infra/jenkins/Jenkinsfile.api BFF cliente + admin-api
softmusic-web infra/jenkins/Jenkinsfile.web app + LP + admin-web via NGINX

Exemplo — uso de credentials() no Jenkinsfile

groovy — trecho Jenkinsfile.api 
environment {
  REGISTRY        = 'docker.io'
  IMAGE_API       = 'softmusic-api'
  DEPLOY_USER     = 'deploy'
  DEPLOY_HOST     = credentials('softmusic-deploy-host')
  DEPLOY_SSH_PORT = credentials('softmusic-deploy-host-port')
  DOCKER_CREDS    = credentials('softmusic-docker-registry-creds')
}

stage('Enviar .env') {
  steps {
    withCredentials([file(credentialsId: 'softmusic-production-env', variable: 'ENV_FILE')]) {
      sshagent(credentials: ['softmusic-ssh-deploy-vps']) {
        sh '''
          scp -o StrictHostKeyChecking=no -P "$DEPLOY_SSH_PORT" "$ENV_FILE" \
            "$DEPLOY_USER@$DEPLOY_HOST:/opt/softmusic/.env.production"
        '''
      }
    }
  }
}
  • DOCKER_CREDS expõe DOCKER_CREDS_USR e DOCKER_CREDS_PSW automaticamente.
  • Infra usa withCredentials([file(...)]) para o .env; jobs de app fazem o mesmo antes do deploy remoto.
  • Nenhum segredo hardcoded — apenas IDs de credential conhecidos.

Guia de escolha — MySQL 8.4 vs 5.7 (legacy)

CenárioJobMySQL
VPS nova, CPU x86_64 moderna (AVX2) softmusic-infra 8.4 (padrão)
CPU antiga, erro ao subir MySQL 8.x, sem suporte a instruções recentes softmusic-infra-legacy 5.7 (LEGACY_MYSQL=1)
Escolha uma vez Rode apenas um dos jobs de infra no primeiro deploy. Não alterne entre 8.4 e 5.7 no mesmo volume de dados sem migração planejada.

Sintomas de incompatibilidade na VPS antiga: container MySQL em restart loop, logs com Illegal instruction ou falha ao iniciar mysqld. Nesse caso, use Jenkinsfile.infra-legacy.

bash — verificar na VPS 
# Após deploy infra (moderno)
docker ps --filter name=softmusic-mysql --format 'table {{.Names}}\t{{.Status}}'
docker exec softmusic-mysql mysqladmin ping -h localhost --silent && echo "MySQL OK"

# Legacy — smoke do Jenkinsfile.infra-legacy
docker exec softmusic-mysql mysqladmin ping -h localhost --silent && echo "MySQL 5.7 OK"

Parte 5 — Ordem do primeiro deploy

1. VPS + clone → 2. Credenciais → 3. infra ou infra-legacy → 4. ia → 5. api → 6. web → 7. DNS
  1. Preparar VPS: Docker, usuário deploy, UFW 80/443, clone em /opt/softmusic/repo
  2. Cadastrar as 5 credenciais no Jenkins
  3. Rodar softmusic-infra ou softmusic-infra-legacy (não os dois)
  4. Rodar softmusic-ia — requer GPU na mesma VPS (ou adapte para worker remoto)
  5. Rodar softmusic-api — build, push e deploy da API
  6. Rodar softmusic-web — build/push de web + LP, deploy com NGINX
  7. Configurar DNS: softmusic.com.br, app.softmusic.com.br, admin.softmusic.com.br → IP da VPS
Deploys seguintes Após o primeiro go-live, rode apenas o job da camada alterada (ex.: só softmusic-api após mudança no BFF).

Parte 6 — Smoke tests (curl)

Execute na VPS (como deploy) ou de uma máquina com acesso aos domínios.

6.1 Na VPS — localhost

bash — VPS produção 
# Infra
docker ps --filter name=softmusic-mysql --filter name=softmusic-redis --filter name=softmusic-rabbitmq \
  --format 'table {{.Names}}\t{{.Status}}'
docker exec softmusic-mysql mysqladmin ping -h localhost --silent && echo "MySQL OK"

# IA
curl -sf http://127.0.0.1:8000/health && echo " Python-AI OK"

# API
curl -sf http://127.0.0.1:8080/health/live && echo " API OK"

# Web (container interno)
curl -sf http://127.0.0.1:5173/ && echo " Web OK"

# NGINX
curl -sf -o /dev/null -w '%{http_code}\n' http://127.0.0.1:80

6.2 Externo — domínios públicos (HTTPS)

bash — qualquer host 
# Landing page
curl -sfI https://softmusic.com.br/ | head -5

# App + API health
curl -sf https://app.softmusic.com.br/api/health/live
curl -sf https://app.softmusic.com.br/api/health/ready

# Admin (espera 200 ou redirect para login)
curl -sfI https://admin.softmusic.com.br/ | head -5

Parte 7 — Troubleshooting

SintomaCausa provávelSolução
CredentialsNotFoundException ID de credential diferente do Jenkinsfile Conferir os 5 IDs exatos da seção 3
Permission denied (publickey) Chave SSH não autorizada Refazer passo 2.2; chmod 600 em authorized_keys
ERRO: clone o repositório em /opt/softmusic/repo Primeiro deploy sem clone Executar seção 1.6 antes do job infra
MySQL container restart loop CPU incompatível com MySQL 8.4 Usar softmusic-infra-legacy (MySQL 5.7)
unauthorized no docker push PAT Docker Hub inválido Atualizar softmusic-docker-registry-creds com PAT Read & Write
docker.sock permission denied deploy fora do grupo docker usermod -aG docker deploy e novo login SSH
IA health check falhou GPU / NVIDIA Toolkit ausente Seção 1.7; nvidia-smi no host e no container
502 / connection refused nos domínios NGINX ou app não subiu docker logs softmusic-nginx; rodar softmusic-web novamente
API OK local, falha no domínio DNS ou TLS Verificar registros A/CNAME e certificados no NGINX
Webhook Asaas não chega URL ou token incorreto ASAAS_WEBHOOK_TOKEN no .env; URL pública /api/webhooks/asaas
